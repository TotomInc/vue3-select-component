The assembled `Select` is not a separate implementation. It is a fixed composition of primitives with extra glue for common app data shapes, events, and form attributes.

If you already use `Select` in your app, this page shows what happens under the hood and where to customize when defaults are not enough.

## Same tree as primitives

Assembled `Select` renders the same primitive tree described in [Primitives](/guide/primitives). `SelectRoot` owns state. Child primitives read it through context.

```
Select (assembled)
└── SelectRoot
    ├── SelectTrigger
    │   ├── SelectIcon (optional slot)
    │   ├── SelectValue
    │   ├── SelectInput (when searchable)
    │   ├── SelectTrailingIcon
    │   └── SelectClear (when clearable)
    └── SelectPopover
        └── SelectListbox
            ├── SelectNoOptions
            ├── SelectOption (×N, from options prop)
            └── SelectCreateItem (when createItem is set)
```

The assembled component wires every slot and prop to the matching primitive. See [Slots](/api/slots) for the mapping.

## What SelectRoot provides

`SelectRoot` is the only place that holds select state. It runs two composables:

| Layer | Role |
|-------|------|
| `useSelectCollection` | Merges options from the `options` prop and declarative `SelectOption` children |
| `useSelectState` | Owns `v-model`, open state, search, filtering, keyboard navigation, and selection |

The result is injected as `SelectContext`. Every primitive calls `injectSelectContext()` to read or update shared state. Primitives outside `SelectRoot` throw at runtime.

Relevant context fields:

- **Selection**: `modelValue`, `select`, `deselect`, `clear`
- **Menu**: `isOpen`, `open`, `close`, `toggle`
- **Search**: `searchValue`, `filteredOptions`
- **Keyboard**: `activeOptionValue`, `moveActiveOption`, `handleKeydown`
- **DOM refs**: `registerTriggerElement`, `registerInputElement`, and IDs for ARIA wiring

Assembled `Select` passes behavior props (`multiple`, `searchable`, `filterBy`, and the rest) straight to `SelectRoot`. It does not reimplement any of this logic.

## Assembled layer glue

### Option normalization

Your `options` prop can use any shape when you pass `getOptionValue` and `getOptionLabel`. Assembled `Select` normalizes them to `{ label, value, disabled? }[]` before passing them to `SelectRoot`.

It also renders one `SelectOption` per normalized entry. That keeps keyboard navigation, filtering, and ARIA in sync with what you passed in.

### filterBy adapter

On primitives, `filterBy` receives a registered option and the search string. On assembled `Select`, `filterBy` receives your **source** option object, its label, and the search string.

The adapter maps values back to source objects so custom filters work with your data model without touching `SelectRoot`.

### Event payloads

`SelectRoot` emits raw values on `@option-selected` and `@option-deselected`. Assembled `Select` looks up the matching source object and re-emits it as `@option-selected` / `@option-deselected`.

This matches v0 behavior and keeps TypeScript tied to your option type.

### Attribute forwarding

Assembled `Select` sets `inheritAttrs: false` and splits fallthrough attributes:

| Attribute group | Searchable (`true`, default) | Non-searchable |
|-----------------|------------------------------|----------------|
| `id`, `name`, `form`, `required`, `aria-*` | `SelectInput` | `SelectTrigger` |
| Everything else | `SelectRoot` | `SelectRoot` |

The focusable control owns labelling attributes. See [Accessibility](/guide/accessibility#label-your-select) for examples.

### Popover props

Props that belong to positioning and teleport (`teleport`, `side`, `align`, and other [reka-ui Popover](https://reka-ui.com/docs/components/popover) options) are stripped from root props and forwarded to `SelectPopover`.

Search and filtering follow the same path. Typing updates `searchValue` in context. `filteredOptions` recomputes. `SelectListbox` only shows options that pass the filter.

## Defaults

These are assembled-only conveniences. Primitives do not implement them.

| Feature | Assembled default | Primitive equivalent |
|---------|-------------------|----------------------|
| Search | `searchable: true` | Add `SelectInput`, pass `searchable` to `SelectRoot` |
| Options list | `v-for` over normalized options | Render `SelectOption` yourself or rely on `options` prop |
| Empty state | `SelectNoOptions` wired | Add `SelectNoOptions` in your listbox |
| Clear button | `clearable` prop + `SelectClear` | Render `SelectClear` when needed |
| Create option | `createItem` prop + `SelectCreateItem` | Add `SelectCreateItem` manually |
| Styles | `data-assembled-select` + import `styles` | Use `data-select-*` hooks on each primitive |

## When to drop to primitives

Stay on assembled `Select` when the default trigger, listbox, and popover layout fit your product.

Reach for primitives when you need to:

- Change DOM structure (infinite scroll, virtual lists, custom groups)
- Own open state or popover rendering
- Emit raw values instead of source objects
- Skip option normalization and register options only via children

Start from the [Primitives](/guide/primitives) guide or the [examples](/guide/examples#primitives) page. Copy the assembled tree and replace the pieces you need.
